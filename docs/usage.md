# Boom Usage Guide

This guide documents every Boom command, when to use it, important options, and the artifacts it produces.

## Command Overview

```bash
boom run [rpc]
boom probe <rpc>
boom catalog <rpc>
boom bench <rpc> [workload options]
boom live <rpc> [workload options]
boom find-limit <rpc> [workload options]
boom ws-bench <ws-rpc>
boom compare <left-rpc> <right-rpc> [workload options]
boom report --run <run-dir>
boom metrics --run <run-dir>
boom engine --rpc <engine-rpc> --jwt <jwt-secret-or-file>
boom engine-ssz-suite --base <engine-rest-base> --jwt <jwt-secret-or-file>
boom engine-ssz --base <engine-rest-base> --jwt <jwt-secret-or-file> --endpoint <endpoint>
```

Use `boom <command> --help` for the exact CLI help generated by the binary.

## `boom run`

Run starts a colored interactive menu for common workflows.

Use it when:

- You want to drive Boom without memorizing flags.
- You are exploring a new endpoint manually.
- You want guided choices for probe, catalog, bench, report, saturation, or WebSocket runs.

Examples:

```bash
boom run
boom run http://localhost:8545
```

Menu workflows:

| Choice | Workflow |
|---|---|
| `1` | Probe endpoint health. |
| `2` | Catalog supported methods. |
| `3` | Run ETH benchmark. |
| `4` | Run comprehensive benchmark. |
| `5` | Open or generate report. |
| `6` | Find saturation limit. |
| `7` | Run WebSocket benchmark. |

`boom run` requires an interactive terminal. For scripts, use the direct subcommands.

## Installation And Development

Install the binary:

```bash
make install
```

Run without installing:

```bash
cargo run -q -- bench http://localhost:8545 --eth
```

Run project checks:

```bash
make ci
```

## `boom probe`

Probe checks basic endpoint health, chain metadata, and method availability.

Use it when:

- You have a new RPC URL and want a quick sanity check.
- You want to identify client version, chain ID, and latest block availability.
- You want to confirm basic JSON-RPC connectivity before benchmarking.

Examples:

```bash
boom probe http://localhost:8545
boom probe --rpc http://localhost:8545
boom probe http://localhost:8545 --timeout 5s
boom probe http://localhost:8551 --jwt ./jwt.hex
```

Important options:

| Option | Purpose |
|---|---|
| `<RPC>` | Positional JSON-RPC URL. |
| `--rpc <RPC>` | Named JSON-RPC URL. Useful in scripts. |
| `--jwt <SECRET_OR_FILE>` | Adds Engine-style bearer JWT auth. |
| `--timeout <DURATION>` | Request timeout, for example `5s`, `500ms`, `1m`. |

Output:

- Prints JSON to stdout.
- Includes client version, chain ID, block number, and per-method probe statuses.

## `boom catalog`

Catalog probes a broad set of JSON-RPC methods and reports whether each method is supported, unsupported, auth-gated, invalid for the provided params, or failing.

Use it when:

- You are onboarding a new RPC endpoint and want a compatibility map.
- You are checking non-standard devnet, Kurtosis, or private RPC methods.
- You want a fast way to compare supported namespaces before benchmarking.

Examples:

```bash
boom catalog http://localhost:8545 --all
boom catalog http://localhost:8545 --methods reth_getRawReceipts,debug_accountRange --json
boom catalog http://localhost:8551 --jwt ./jwt.hex --all
```

Important options:

| Option | Purpose |
|---|---|
| `<RPC>` | Positional JSON-RPC URL. |
| `--rpc <RPC>` | Named JSON-RPC URL. |
| `--jwt <SECRET_OR_FILE>` | Adds bearer JWT auth for Engine/private endpoints. |
| `--all` | Include heavier debug, trace, txpool, and Engine probes. |
| `--methods <A,B>` | Additional comma-separated method names to probe. |
| `--json` | Print structured JSON instead of CSV-style terminal output. |

Output:

- Terminal table by default: `method,status,latency_ms,detail`.
- JSON report with endpoint metadata and per-method probe records when `--json` is set.

## `boom bench`

Bench runs a single-endpoint benchmark and writes structured artifacts.

Use it when:

- You want to measure one RPC endpoint under load.
- You want fixed-rate, ramped-rate, batch, or concurrency-driven results.
- You want a report for a single endpoint.
- You want repeatable output artifacts for later comparison.

Basic examples:

```bash
boom bench http://localhost:8545 --eth --duration 30s --out runs/local-eth
boom bench http://localhost:8545 --all --duration 120s --concurrency 512 --out runs/heavy
boom bench http://localhost:8545 --scenario explorer --out runs/explorer
```

Automation examples:

```bash
boom bench http://localhost:8545 --all --no-prompt --out runs/ci
boom bench http://localhost:8545 --all --json --out runs/ci-json
```

### Workload Selection

Preset flags are additive:

| Option | Use When |
|---|---|
| `--eth` | You want normal Ethereum JSON-RPC read traffic. |
| `--debug` | You want debug tracing load such as `debug_traceTransaction`. |
| `--trace` | You want OpenEthereum/Erigon-style trace namespace coverage. |
| `--txpool` | You want transaction pool namespace coverage. |
| `--net` | You want network metadata calls. |
| `--web3` | You want web3 namespace calls. |
| `--all` | You want every built-in suite. |

If no preset or scenario is provided, Boom defaults to the ETH workload.

Scenario shortcuts:

| Scenario | Use When |
|---|---|
| `light` | You want a small, cheap smoke benchmark. |
| `explorer` | You want explorer-like block, receipt, and log reads. |
| `archive` | You want balance/code/storage/call style archive reads. |
| `debug-heavy` | You want tracing-heavy workload. |
| `simulate` | You want `eth_simulateV1` only. This is opt-in because simulation APIs are more client/version-sensitive than basic ETH reads. |
| `txpool` | You want transaction-pool-focused traffic. |
| `all` | You want all built-in suites. |

Examples:

```bash
boom bench http://localhost:8545 --scenario light
boom bench http://localhost:8545 --scenario explorer --duration 2m
boom bench http://localhost:8545 --scenario debug-heavy --timeout 30s
boom bench http://localhost:8545 --scenario simulate --timeout 30s
```

### Load Control

Concurrency-driven mode:

```bash
boom bench http://localhost:8545 --all --concurrency 256 --duration 60s
```

Fixed request-rate mode:

```bash
boom bench http://localhost:8545 --all --rps 500 --duration 60s
```

Ramped request-rate mode:

```bash
boom bench http://localhost:8545 --all --ramp 100:1000 --duration 5m
```

Batch mode:

```bash
boom bench http://localhost:8545 --eth --batch-size 20 --rps 500 --duration 60s
```

Important options:

| Option | Purpose |
|---|---|
| `--duration <DURATION>` | Main measured duration. |
| `--warmup <DURATION>` | Warmup phase before measurements. |
| `--concurrency <N>` | Number of concurrent workers. |
| `--timeout <DURATION>` | Per-request timeout. |
| `--batch-size <N>` | Number of JSON-RPC calls per HTTP batch request. |
| `--rps <N>` | Fixed request rate target. |
| `--ramp <START:END>` | Linear request-rate ramp over the run duration. |
| `--out <DIR>` | Output directory. |
| `--json` | Print machine-readable JSON summary to stdout. |
| `--no-prompt` | Disable interactive report prompts. |

### Config File Mode

Use config files when you need exact method weights or custom params.

```bash
boom bench --config configs/examples/mini-eth.toml --out runs/mini-eth
boom bench --config configs/examples/basic-eth.toml --out runs/basic
```

Config mode ignores preset flags because the TOML file is the workload source. See the [config guide](config.md) for full `config.toml` structure, fields, placeholders, and examples.

### Benchmark Artifacts

Each bench run writes:

| File | Meaning |
|---|---|
| `seed.json` | Live seed data fetched from the RPC. |
| `run.json` | Complete structured summary. |
| `metrics.csv` | Per-method aggregate metrics. |
| `samples.csv` | Per-second time-series metrics. |
| `errors.jsonl` | Capped error samples for debugging. |

Boom resolves live placeholders before the run:

| Placeholder | Meaning |
|---|---|
| `$latest_block` | Latest block number. |
| `$block_hash` | Latest block hash. |
| `$tx_hash` | Transaction hash from a seeded block. |
| `$address` | Address found from seeded transaction data or zero address fallback. |
| `$call_to` | Call target found from seeded transaction data or fallback. |

If a placeholder cannot be resolved, methods depending on it are skipped instead of sending fake data.

When running interactively, `boom bench` prompts by default with yes as the default answer for terminal summary and HTML report generation. Use `--no-prompt` for CI or unattended runs.

## `boom compare`

Compare runs the same workload against two endpoints and writes a comparison report.

Use it when:

- You want to compare two clients, such as reth vs geth.
- You want to compare two node configurations.
- You want to compare local node performance against a provider.
- You want per-method p95 and success-rate deltas.

Examples:

```bash
boom compare http://localhost:8545 http://localhost:9545 --eth --duration 60s --out runs/compare-eth
boom compare http://localhost:8545 http://localhost:9545 --all --rps 500 --out runs/compare-rate
boom compare http://localhost:8545 http://localhost:9545 --scenario explorer --batch-size 20 --open
```

Important options:

| Option | Purpose |
|---|---|
| `<LEFT>` | Left endpoint URL. |
| `<RIGHT>` | Right endpoint URL. |
| `--out <DIR>` | Compare output directory. |
| workload options | Same workload/load options as `boom bench`. |
| `--open` | Open `compare.html` after generation. |

Compare artifacts:

| File/Directory | Meaning |
|---|---|
| `left/` | Full benchmark artifacts for the left endpoint. |
| `right/` | Full benchmark artifacts for the right endpoint. |
| `compare.json` | Structured comparison summary. |
| `compare.md` | Markdown comparison report. |
| `compare.html` | HTML comparison report. |

## `boom live`

Live runs the normal benchmark engine while keeping a terminal status line active. At completion it prints the detailed terminal summary and generates the report artifacts.

Use it when:

- You are running a long benchmark interactively.
- You want a quick terminal dashboard without opening the HTML report.
- You want the same artifacts as `boom bench` with more operator feedback.

Examples:

```bash
boom live http://localhost:8545 --scenario explorer --duration 5m --out runs/live
boom live http://localhost:8545 --all --rps 500 --duration 2m --out runs/live-rate
```

`boom live` accepts the same workload and load-control options as `boom bench`.

## `boom find-limit`

Find-limit is a saturation finder. It runs repeated fixed-rate benchmarks, increasing RPS until the endpoint breaks the latency or error budget.

Use it when:

- You want a defensible maximum sustainable RPS for a node.
- You need a capacity number for release validation.
- You want to compare configurations using latency and error budgets instead of raw throughput alone.

Examples:

```bash
boom find-limit http://localhost:8545 --scenario explorer --target-p95 250 --max-error-rate 1 --out runs/limit
boom find-limit http://localhost:8545 --all --start-rps 100 --max-rps 5000 --step 1.4 --duration 30s
```

Important options:

| Option | Purpose |
|---|---|
| `<RPC>` | Endpoint to test. |
| `--out <DIR>` | Root output directory for all attempts. |
| `--start-rps <N>` | First requested RPS, default `50`. |
| `--max-rps <N>` | Upper search bound, default `5000`. |
| `--step <N>` | Multiplicative RPS increase, default `1.5`. |
| `--target-p95 <MS>` | Latency budget, default `250`. |
| `--max-error-rate <PCT>` | Error budget, default `1.0`. |
| workload options | Same scenario/preset/duration/concurrency options as `boom bench`. |

Artifacts:

| File/Directory | Meaning |
|---|---|
| `rps-*/` | Full benchmark artifacts for each attempted RPS. |
| `limit.json` | Structured saturation result. |
| `limit.md` | Human-readable saturation table. |

## `boom ws-bench`

WebSocket bench sends JSON-RPC calls over a WebSocket endpoint and writes the same `run.json` shape used by HTTP benchmarks.

Use it when:

- Your node exposes JSON-RPC over WebSocket.
- You want to compare HTTP and WebSocket transport behavior.
- You want to validate WebSocket latency, timeout, and error behavior under concurrency.

Examples:

```bash
boom ws-bench ws://localhost:8546 --method eth_blockNumber --duration 30s --out runs/ws
boom ws-bench ws://localhost:8546 --method eth_getBalance --params '["0x0000000000000000000000000000000000000000","latest"]'
```

Important options:

| Option | Purpose |
|---|---|
| `<WS>` | WebSocket JSON-RPC URL. |
| `--duration <DURATION>` | Measured duration. |
| `--concurrency <N>` | Number of WebSocket workers. |
| `--timeout <DURATION>` | Per-request timeout. |
| `--method <METHOD>` | JSON-RPC method, default `eth_blockNumber`. |
| `--params <JSON>` | JSON params array/object/string, default `[]`. |
| `--out <DIR>` | Output directory. |
| `--json` | Print structured summary to stdout. |

Artifacts:

| File | Meaning |
|---|---|
| `run.json` | Structured benchmark summary. |
| `summary.md` | Markdown summary. |
| `openmetrics.prom` | OpenMetrics text for the run. |

## `boom report`

Report generates readable Markdown and HTML from an existing benchmark run.

Use it when:

- You already have a `run.json` and want a report.
- You want to print a previous run summary.
- You want to open the HTML dashboard for a completed run.

Examples:

```bash
boom report --run runs/mini-eth
boom report --run runs/mini-eth --print
boom report --run runs/mini-eth --open
boom report --run runs/mini-eth --prompt
boom report --run runs/mini-eth --out reports/mini-eth
boom report --run runs/mini-eth --no-prompt
```

Important options:

| Option | Purpose |
|---|---|
| `--run <DIR_OR_RUN_JSON>` | Run directory or direct `run.json` path. |
| `--out <DIR>` | Output directory for generated reports. Defaults to run directory. |
| `--print` | Print Markdown summary to terminal. |
| `--open` | Open generated HTML report in system browser. |
| `--prompt` | Ask whether to print and open. |
| `--no-prompt` | Disable the default interactive prompt. |

Generated files:

| File | Meaning |
|---|---|
| `summary.md` | Human-readable Markdown report. |
| `report.html` | Dark single-file HTML dashboard with inline CSS/SVG, charts, and error reason cards. |

In an interactive terminal, `boom report --run <dir>` prompts by default unless `--print`, `--open`, or `--no-prompt` is provided. The HTML report reads `errors.jsonl` when available and shows grouped error reasons plus individual error samples.

## `boom metrics`

Metrics exports a completed run as Prometheus/OpenMetrics text.

Use it when:

- You want to ingest benchmark results into Prometheus/Grafana.
- You want a CI artifact with stable metric names.
- You want a text format that is easy to diff and archive.

Examples:

```bash
boom metrics --run runs/heavy
boom metrics --run runs/heavy --print
boom metrics --run runs/heavy --out reports/heavy.prom
```

Important options:

| Option | Purpose |
|---|---|
| `--run <DIR_OR_RUN_JSON>` | Run directory or direct `run.json` path. |
| `--out <PATH>` | Output path. Defaults to `openmetrics.prom` in the run directory. |
| `--print` | Print the generated metrics to stdout. |

Metrics include:

- `boom_requests_total`
- `boom_requests_per_second`
- `boom_latency_ms`
- `boom_method_requests_total`
- `boom_method_latency_ms`

## `boom engine`

Engine probes authenticated Engine JSON-RPC capability exchange.

Use it when:

- You want to confirm Engine JSON-RPC auth works.
- You want to verify `engine_exchangeCapabilities` behavior.
- You are testing execution client Engine API availability.

Examples:

```bash
boom engine --rpc http://localhost:8551 --jwt ./jwt.hex
boom engine --rpc http://localhost:8551 --jwt 0123456789abcdef0123456789abcdef --timeout 5s
```

Important options:

| Option | Purpose |
|---|---|
| `--rpc <URL>` | Engine JSON-RPC URL. |
| `--jwt <SECRET_OR_FILE>` | Hex JWT secret or path to JWT secret file. |
| `--timeout <DURATION>` | Request timeout. |

JWT behavior:

- The JWT secret itself does not expire.
- Boom signs a fresh bearer token with a current `iat` timestamp per request.

## `boom engine-ssz-suite`

Engine SSZ suite probes every built-in Engine REST/SSZ route and prints a CSV-style compatibility table.

Use it when:

- You want a quick endpoint map for Engine REST/SSZ support.
- You want to check auth, routing, status codes, content type, and byte counts.
- You are testing execution-apis PR #793 style endpoint support.

Examples:

```bash
boom engine-ssz-suite --base http://localhost:8551 --jwt ./jwt.hex
boom engine-ssz-suite --base http://localhost:8551 --jwt ./jwt.hex --fork prague --accept application/json
```

Important options:

| Option | Purpose |
|---|---|
| `--base <URL>` | Engine REST/SSZ base URL. |
| `--jwt <SECRET_OR_FILE>` | Hex JWT secret or path to secret file. |
| `--fork <NAME>` | Fork path component, default `prague`. |
| `--timeout <DURATION>` | Request timeout. |
| `--accept <MIME>` | Accept header. |

Output columns:

```text
endpoint,status,content_type,bytes
```

## `boom engine-ssz`

Engine SSZ calls one authenticated Engine REST/SSZ endpoint.

Use it when:

- You want to test a single Engine REST/SSZ route.
- You have a fixture body and want to POST it.
- You want precise control over fork, payload ID, body, and Accept header.

Examples:

```bash
boom engine-ssz --base http://localhost:8551 --jwt ./jwt.hex --endpoint capabilities --accept application/json
boom engine-ssz --base http://localhost:8551 --jwt ./jwt.hex --endpoint identity --accept application/json
boom engine-ssz --base http://localhost:8551 --jwt ./jwt.hex --endpoint bodies-by-range --fork prague
boom engine-ssz --base http://localhost:8551 --jwt ./jwt.hex --endpoint new-payload --fork prague --body fixtures/new-payload.ssz
boom engine-ssz --base http://localhost:8551 --jwt ./jwt.hex --endpoint get-payload --payload-id 0x0000000000000000
```

Supported endpoint names:

| Endpoint | Route Type |
|---|---|
| `capabilities` | GET capabilities. |
| `identity` | GET identity. |
| `new-payload` | POST fork payload body. |
| `forkchoice` | POST forkchoice body. |
| `get-payload` | GET payload by payload ID. |
| `bodies-by-hash` | POST body hashes. |
| `bodies-by-range` | GET bodies range. |
| `blobs-v1` | POST blobs v1. |
| `blobs-v2` | POST blobs v2. |
| `blobs-v3` | POST blobs v3. |
| `blobs-v4` | POST blobs v4. |

Important options:

| Option | Purpose |
|---|---|
| `--base <URL>` | Engine REST/SSZ base URL. |
| `--jwt <SECRET_OR_FILE>` | Hex JWT secret or path to secret file. |
| `--endpoint <NAME>` | Endpoint enum value. |
| `--fork <NAME>` | Fork path component, default `prague`. |
| `--payload-id <ID>` | Payload ID for `get-payload`. |
| `--body <PATH>` | Request body file for POST endpoints. |
| `--timeout <DURATION>` | Request timeout. |
| `--accept <MIME>` | Accept header. |

## Output Interpretation

Key top-level metrics:

| Metric | Meaning |
|---|---|
| `total_requests` | Count of logical JSON-RPC calls, not necessarily HTTP requests when batching. |
| `successes` | Calls that returned HTTP success and no JSON-RPC error object. |
| `rpc_errors` | Calls that returned a JSON-RPC error object. |
| `transport_errors` | HTTP/network/client-side failures. |
| `timeouts` | Requests classified as timeout failures. |
| `requests_per_second` | Logical request throughput. |
| `latency` | Aggregate min, p50, p90, p95, p99, and max. |
| `histogram` | Latency bucket counts. |
| `samples` | Per-second request/error/latency samples. |
| `methods` | Per-method request/error/latency summaries. |

## Operational Guidance

Use conservative settings first:

```bash
boom bench http://localhost:8545 --scenario light --rps 50 --duration 30s
```

Increase one dimension at a time:

```bash
boom bench http://localhost:8545 --scenario explorer --rps 250 --duration 60s
boom bench http://localhost:8545 --scenario explorer --rps 500 --duration 60s
boom bench http://localhost:8545 --scenario explorer --ramp 100:1000 --duration 5m
```

Use `--batch-size` separately from `--rps` so you know whether you are testing logical call capacity or HTTP batch handling:

```bash
boom bench http://localhost:8545 --eth --rps 500 --batch-size 1
boom bench http://localhost:8545 --eth --rps 500 --batch-size 20
```

Use `--no-prompt` for automation:

```bash
boom bench http://localhost:8545 --all --rps 500 --duration 60s --no-prompt --out runs/ci
```

## Safety

Only benchmark endpoints you own, operate, or have permission to test. Public endpoints may rate-limit, ban, or return misleading results under load. Start with low `--rps`, short `--duration`, and small `--concurrency`, then scale deliberately.
